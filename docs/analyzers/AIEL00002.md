# AIEL00002 -  General - Error types must have a single string constructor

## Summary
Error types derived from `Error` must have exactly one public constructor that accepts a single `string` parameter representing the error message.

## Problem
```csharp
// ❌ No public constructor
public sealed class ValidationError : Error
{
    private ValidationError(string message) : base(ValidationErrorCode.Instance, message) { }
}

// ❌ Multiple public constructors
public sealed class ConflictError : Error
{
    public ConflictError(string message) : base(ConflictErrorCode.Instance, message) { }
    public ConflictError(string message, int code) : base(ConflictErrorCode.Instance, message) { }
}

// ❌ Wrong parameter type
public sealed class ServerError : Error
{
    public ServerError(int statusCode) : base(ServerErrorCode.Instance, "Server error") { }
}
```

## Solution
Ensure your Error-derived types have exactly one public constructor accepting a single `string` parameter:

```csharp
// ✅ Correct
public sealed class NotFoundError : Error
{
    public NotFoundError(string message) : base(NotFoundErrorCode.Instance, message) { }
    
    public sealed class NotFoundErrorCode : ErrorCode
    {
        public static readonly NotFoundErrorCode Instance = new();
        protected override string Name => "NotFoundError";
    }
}

// ✅ Correct - with init properties
public sealed class OrderNotFoundError : Error
{
    public string OrderId { get; init; }
    
    public OrderNotFoundError(string message) : base(OrderNotFoundErrorCode.Instance, message) { }
    
    public sealed class OrderNotFoundErrorCode : ErrorCode
    {
        public static readonly OrderNotFoundErrorCode Instance = new();
        protected override string Name => "OrderNotFoundError";
    }
}
```

## Why This Matters

### JSON Deserialization
System.Text.Json's parameterized constructor converter requires a clear, unambiguous constructor signature. Having exactly one `(string message)` constructor ensures:
- Proper binding during JSON deserialization
- Predictable behavior across different .NET versions
- Compatibility with Result pattern serialization

### Code Generation
The error code generator validates that custom error types comply with this constraint, ensuring generated code works correctly with the serialization infrastructure.

### Consistency
This requirement enforces a consistent API across all error types in your domain, making error handling predictable.
