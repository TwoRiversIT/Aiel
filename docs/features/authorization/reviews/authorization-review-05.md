## Copilot's Review

Review basis: I reviewed this against your notes and the last committed `permission-system.md` draft because the current working copy of `permission-system.md` is empty.

### Execution Context

I agree with making the concrete execution context immutable and sealed. The interface does not currently buy much, and a mutable `Properties` bag is exactly the sort of extensibility point that becomes a quiet dependency surface before anyone notices. If we later need extensibility, I would rather add named value objects or explicit child-context methods than start with an unbounded dictionary.

I would keep the base name `ExecutionContext` and use `ActionExecutionContext<TAction>` for the typed wrapper. `ActionContext` is a better conceptual name in isolation, but it collides with ASP.NET Core terminology and will make transport code harder to read. The important distinction is:

```csharp
public sealed class ExecutionContext
{
    public OperationId OperationId { get; }
    public IActor Actor { get; }
    public CorrelationId CorrelationId { get; }
    public OperationId? CausationId { get; }
    public ClientInstanceId? ClientInstanceId { get; }
}

public sealed class ActionExecutionContext<TAction>
    where TAction : IAction
{
    public ExecutionContext ExecutionContext { get; }
    public TAction Action { get; }
}
```

That keeps the reusable ambient execution data separate from the action payload without turning the base context into another generic or interface-heavy abstraction.

### Permission-Free Marker

`DoesNotRespectAuthority` is weird in the right direction. I would use it, require a non-empty reason, and make the analyzer warn both where the action is declared and where it is referenced. In Aviendha, a permission-free action is not a convenience flag; it is a safety exception. Making it uncomfortable is a feature.

### Resource-Aware Checkers

The analyzer can enforce concrete checkers only if resource awareness is explicit. It should not try to infer resource-aware actions by guessing from property names or Strong ID types. I would add either a marker interface or metadata attribute, for example:

```csharp
public interface IResourceAction<TResource>
{
    TResource Resource { get; }
}
```

or:

```csharp
[ActionResource(nameof(AppointmentId))]
public sealed record ChangeAppointment(AppointmentId AppointmentId, ...) : ICommand;
```

Once an action is explicitly resource-aware, absence of a concrete `IActionPermissionChecker<TAction>` should be an analyzer error. The generated default checker is fine for flat permissions, but it cannot safely answer resource ownership, assignment, or boundary questions.

### ASP.NET Core Integration

Generated Minimal API endpoints now feel less like an optional convenience and more like the clean transport boundary. They solve the duplicated route-parameter problem you called out because the generated endpoint can own the binding shape and construct the action from route, query, and body inputs without forcing a hand-written controller method to mirror every route token.

I would keep application handlers returning `Result` and `Result<T>`. The generated HTTP adapter should translate failures to `ProblemDetails` at the transport boundary. I would not push `ProblemDetails` into application contracts, and I would not use `Result<ProblemDetails>` as an application-layer return type. That keeps `Aiel.Results` useful everywhere while still giving HTTP clients idiomatic responses.

The tricky design problem is routing metadata. I think v1 needs an explicit route attribute or fluent route map for actions, not convention-only routing. Convention can generate a default, but persisted or public API routes need the same stability discipline as permission names.

### Permission Manifests and Renames

My preferred manifest model is:

1. The source generator emits the current runtime manifest as C#.
2. A design-time tool materializes that generated manifest into a committed snapshot file, probably JSON.
3. The analyzer compares the current generated manifest to the committed snapshot.
4. Any key removal, rename, or scope change requires an explicit migration DSL operation.

The generator should never write the snapshot file directly, so it cannot clobber user-owned files. This is closer to EF Core's model snapshot workflow than to embedded resources. The committed snapshot is the reviewable anchor; the generated manifest is the current compilation truth.

For rename detection, the analyzer can detect that a permission disappeared and another appeared, but it cannot always know whether that is a rename or an intentional remove/add. We probably need an explicit migration DSL operation to declare intent:

```csharp
migrationBuilder.Permissions()
    .Rename(
        from: "Aviendha.Scheduling.ChangeAppointment",
        to: AviendhaAuthorization.Scheduling.Appointments.ChangeAppointment);
```

The analyzer's job is to block unmanaged drift, not to infer every migration automatically.

### Store, Manager, and Decisions

I agree that returning `AuthorizationGrantDecision` from the store smells wrong. The store should return persisted grant state only. The evaluator should produce the decision after considering grant state, prohibition precedence, lifecycle, scope chain, subject resolution, and fail-closed defaults.

Because Aiel public APIs should not expose nullable return values, I would avoid `Result<PermissionGrantRecord?>`. Better shapes would be either an explicit optional value object or a lookup result:

```csharp
public abstract record PermissionGrantLookup;

public sealed record PermissionGrantFound(
    PermissionGrantRecord Grant) : PermissionGrantLookup;

public sealed record PermissionGrantNotFound : PermissionGrantLookup;
```

or, preferably, a first-class optional pattern if Aiel settles one. The important point is that absence of a grant is data, not an exception and not a null.

On decorators, SimpleInjector is worth studying, especially its deterministic decorator model, but I would be cautious about taking a dependency in core Aiel. A small native decorator registration story may be better long term, with SimpleInjector/Rebus.SimpleInjector used as prior art.

### `IPermissionSubjectValidator`

This should be the application-owned bridge between generic permission grants and application identity. The permissions EF package can store `PermissionSubject`, but it should not know how to validate a user, role, service principal, clinic actor, or tenant-local subject.

I would shape validators around subject type ownership:

```csharp
public interface IPermissionSubjectValidator
{
    PermissionSubjectType SubjectType { get; }

    ValueTask<Result> ValidateAsync(
        PermissionSubject subject,
        PermissionScope scope,
        CancellationToken cancellationToken = default);
}
```

Including `PermissionScope` is useful because a subject may exist globally but not be grantable in a particular tenant, clinic, or host scope. The manager can resolve the validator by `SubjectType`, validate before writes, and fail closed when no validator exists for an application-defined subject type.

### Client-Side Capability and Blazor

For v1, I agree with service-based Blazor support and no resource-specific client-side permission execution. Resource-specific availability should come from server query results when the screen already has to load the resource. Coarse availability can come from a capability snapshot endpoint for navigation and top-level affordances.

The client-side `IAuthorizationChecker` should be a read model over the latest snapshot and any DTO-embedded availability, not a second authorization engine. A failed server authorization response should invalidate the relevant snapshot and trigger a refresh. Fluxor could host that state nicely, but the permission package should not require Fluxor in the base client abstractions.

I would not try to mirror ASP.NET Core policies in Blazor for v1. `AuthorizeView` is good for coarse authentication and role/policy UI, but nested context management gets awkward quickly, and resource-aware permissions need data the policy system does not naturally have. A direct `if` against `IAuthorizationChecker` is boring in a good way.

### Action Verb Diagnostics

I agree with the unified attribute shape:

```csharp
[ActionVerb(Disallow = "Update")]
[ActionVerb(Disallow = "Manage")]
[ActionVerb(Allow = "Triage")]
[ActionVerb(Allow = "Refer")]
[ActionVerb(Allow = "Enroll")]
public sealed partial class AviendhaActionVocabulary;
```

The analyzer should ship with conservative defaults and let applications tighten them. The goal is not to win a naming argument; it is to prevent vague action names from becoming vague permission names.

### Net Recommendation

I think the design is ready to move from exploratory draft to an implementation plan, with three decisions made explicit before code starts:

1. `ExecutionContext` and `ActionExecutionContext<TAction>` are immutable sealed framework types.
2. Resource-aware actions must declare resource awareness explicitly and must have concrete checkers.
3. Generated Minimal API endpoints are the preferred ASP.NET v1 adapter, with `Result` mapped to `ProblemDetails` only at the HTTP boundary.

The biggest remaining risk is migration discipline around generated permission names. The manifest snapshot plus explicit migration DSL is the piece that has to be boring and strict; if that is solid, the rest of the action-centered model fits Aiel well.

## Doug's Turn

### Resource-Aware Checkers

I do not like the `IResourceAction<TResource>` marker interface, specifically the `TResource Resource { get; }` property. We do not want commands or queries carrying the resource(s) they are targeting, and because it is an interface, it requires an implementation which would allow exactly that. 

The marker attribute is a little more palatable, but it still requires the developer to be disciplined about applying it consistently, and because it exists to support the analyzer, it will be difficult to detect when it should be present but is not.

How about:

```csharp
abstract record ResourceCommand<TResource> : ICommand;
```

NOTE: I am not sure when `Resource` got introduced, but it is a bit misleading when assemblies can have embedded resources, and while I have not had much use for them, `.resx` files are a thing. In other systems I have developed I used `Entity` because that is what DDD uses.

### ASP.NET Core Integration

> I would keep application handlers returning `Result` and `Result<T>`. The generated HTTP adapter should translate failures to `ProblemDetails` at the transport boundary. I would not push `ProblemDetails` into application contracts, and I would not use `Result<ProblemDetails>` as an application-layer return type. That keeps `Aiel.Results` useful everywhere while still giving HTTP clients idiomatic responses.

I have thought about this some more and I have to disagree. `Aiel.Results` has a robust error code system that easily surpasses `ProblemDetails`, and any actions that come in from sources other than HTTP already rely on it. To be consistent, the client should too. In fact, `Result<T>` will be returned from any client side services. We are committed to clean architecture and putting our application service interfaces in `*.Contracts` packages so they can be easily consumed by clients regardless of where they run.

I also keep forgetting that clients other than Aviendha might be Blazor Server or Blazor WebApps and could conceivably inject the application services and consume them directly.

And now that I think about that, I realize that we are losing sight of true CQRS. We are getting caught up in slick pipelines and pipeline behaviours which terminate in handlers, not application services. I was not sure why you were calling out routing metadata as a tricky problem because if we are generating both the HTTP endpoints and the HTTP client we can do whatever is convenient. But that is based on the assumption that the HTTP endpoint would just fire the action into the pipeline and then return the result. 

I agree with most of the rest of your review, but I want to have your input on this before I commit to transitioning to an implementation plan.
